# Editing Obsidian In This Workspace

This note explains how Obsidian behavior has been changed in this vault, where those changes live, and how to edit them without losing work the next time the vault is refreshed or bootstrapped.

The big idea: this workspace is one Obsidian vault. The root `.obsidian` folder is the active profile. The `_system/` folder documents and regenerates the vault operating system and now also holds reusable tools, scripts, assets, and agent skills.

## Runtime Vs Source

There are three different kinds of Obsidian edits here.

### 1. Runtime Profile Edits

These are changes in the active root profile:

```text
.obsidian/
```

Examples:

- `.obsidian/appearance.json`
- `.obsidian/community-plugins.json`
- `.obsidian/hotkeys.json`
- `.obsidian/snippets/*.css`
- `.obsidian/plugins/*/data.json`
- `.obsidian/themes/<current-theme>/theme.css`

This is what Obsidian is actually using right now.

Root `.obsidian` is the source of truth for Obsidian settings. Bootstrap does not copy or patch it.

If you want a backup first, run:

```bash
vault backup
```

### 2. Generated Vault Files

Many useful Obsidian files live in the vault itself and are regenerated by scripts:

```text
_system/_obsidian/bases/
<context-folder>/_obsidian/bases/
<context-folder>/_obsidian/periodic/
_system/_obsidian/periodic/
```

Generated files usually include a managed marker such as:

```text
vault.bootstrap
```

If a file is generated, prefer changing the generator instead of editing the output.

Important generators:

- `_system/commands/bootstrap_export.py`: public vault export from the current vault.
- `_system/bootstrap/bootstrap_vault.py`: scaffold/reconcile script and many default Base/template files.
- `_system/commands/refresh.py` and `dashboard.py`: refresh orchestration and generated Dashboard.
- `_system/commands/periodic.py`: daily/weekly/quarterly/yearly rollups.
- `_system/commands/content.py`: content schedule notes and content Bases.
- `_system/commands/epic.py`: epic notes and per-epic TaskNotes Kanban Bases.

Use `vault --help` to see the command wrapper.

### 3. Custom Plugin Code

The custom plugin is not authored inside the vault. It is developed here:

```text
`context-nine-obsidian-plugin` path from topology repository config
```

It is installed into the active vault here:

```text
.obsidian/plugins/context-nine/
```

To edit it:

```bash
cd "<configured context-nine-obsidian-plugin path>"
npm run build
npm run test
npm run install-vault
```

Then reload Obsidian or disable/enable the plugin.

The useful source files are:

- `src/main.ts`: plugin startup, commands, settings tab.
- `src/tasknotes-ux.ts`: TaskNotes natural-language trigger patches, priority aliasing, epic picker in modals.
- `src/task-capture.ts`: capture selection to new/existing TaskNotes tasks.
- `src/file-actions.ts`: hovered/selected file deletion, Bases card deletion, TaskNotes modal delete button.
- `src/attachment-router.ts`: attachment inbox routing.
- `src/vault-cockpit.ts`: right sidebar cockpit for running `vault` commands.
- `styles.css`: plugin UI styling, mostly for its own modals/cockpit.

Context Nine does not render the TaskNotes Kanban board. It patches surrounding UX and adds commands.

## What Was Changed For TaskNotes And Kanban

The task board changes are a stack:

### TaskNotes Plugin Settings

TaskNotes stores key settings here:

```text
.obsidian/plugins/tasknotes/data.json
```

Important fields:

- `fieldMapping.sortOrder`: currently maps manual ordering to `tasknotes_manual_order`.
- `customStatuses`: status values such as `backlog`, `up-next`, `to-be-resumed`, `ongoing`, `in-progress`, `done`, `archived`.
- `customPriorities`: priority values such as `none`, `low`, `normal`, `high`.
- task creation defaults and visible properties.

Use TaskNotes native fields in task frontmatter:

```yaml
status: up-next
priority: normal
scheduled: 2026-05-15
due: 2026-05-20
timeEstimate: 60
tasknotes_manual_order: "0|i00007:"
```

Do not invent alternate fields like `due_date`, `scheduled_date`, or `duration`.

### TaskNotes Bases Views

The main TaskNotes Kanban Base is:

```text
_system/_obsidian/bases/tasks-kanban-v1.base
```

Most views use:

```yaml
type: tasknotesKanban
groupBy:
  property: status
  direction: ASC
swimLane: note.projects
sort:
  - column: tasknotes_manual_order
    direction: DESC
```

That means:

- columns are task statuses;
- swimlanes are projects;
- manual drag order is stored in each task note's `tasknotes_manual_order` frontmatter;
- dragging across columns updates `status`;
- dragging within a manual-order view should persist by changing `tasknotes_manual_order`.

The generated epic views in this file are managed by:

```text
_system/commands/epic.py
```

If you add, rename, or delete epics, use:

```bash
vault epic sync
```

### TaskNotes Swimlane Styling

The modern swimlane/card styling is CSS, not plugin logic:

```text
.obsidian/snippets/tasknotes-swimlanes.css
```

It is enabled in:

```text
.obsidian/appearance.json
```

Edit this snippet when changing:

- card rounded corners, spacing, hover states;
- swimlane row layout;
- sticky swimlane/column headers;
- status lane colors;
- TaskNotes Kanban visual density.

After changing it, root `.obsidian` already contains the live styling source.

### Two Kanban Systems

There are two different Kanban implementations in this vault:

1. `type: tasknotesKanban`

This is TaskNotes' own Bases view. Use it for TaskNotes task boards. It persists card order through `tasknotes_manual_order` in task frontmatter.

2. `type: kanban-view`

This comes from the `kanban-bases-view` community plugin. Use it for content boards and general Bases boards. It persists card/column/swimlane order inside the `.base` file using fields such as `cardOrders`, `columnOrders`, and `swimlaneOrders`.

Do not mix their persistence models. If task ordering should be visible to agents and survive as task data, use TaskNotes' `tasknotes_manual_order`.

## Content Calendar And Content Kanban

Content infrastructure lives under content-enabled context folders:

```text
personal-brand/_obsidian/content/
business/_obsidian/content/
```

The vault rollups are:

```text
_system/_obsidian/bases/content-calendar.base
_system/_obsidian/bases/content-kanban.base
```

Content calendars use the `calendar-bases` plugin and the `publish_date` property.

Content Kanban uses `kanban-bases-view`, not TaskNotes. Status is stored in content note frontmatter, and the board is configured in the `.base` file.

The content source of truth is `_obsidian/content` note frontmatter. Do not convert content items into TaskNotes tasks just so they appear on a board.

## Attachments

Obsidian's built-in attachment target is deliberately set to:

```text
_system/_obsidian/attachments/_inbox
```

The vault convention is that attachments should end up under the owning root folder:

```text
<root-folder>/_obsidian/attachments/
```

Use:

```bash
vault attachments
vault attachments --apply
vault attachments --verify-only
```

Implementation:

```text
_system/commands/attachments.py
```

Context Nine also routes ordinary pasted/dropped inbox attachments in the background when enabled.

## Editing Checklist

Before editing:

1. Decide whether the behavior is profile config, generated vault files, CSS/theme, TaskNotes data, a community plugin setting, or custom plugin code.
2. Check [[_system/docs/obsidian/README|Obsidian Profile]] and [[_system/docs/commands/README|Scripts README]].
3. If a file has a managed marker, edit the generator instead.
4. If changing a plugin setting through Obsidian, verify which `.obsidian/plugins/<plugin-id>/data.json` file changed.
5. If changing Context Nine, resolve `context-nine-obsidian-plugin` through topology config, then build/install.

After editing:

1. Test in Obsidian.
2. Run relevant generator command after generated-file changes, such as `vault refresh`, `vault content`, or `vault epic sync`.
3. Update this note, [[_system/docs/obsidian/README|Obsidian Profile]], or [[_system/README|System README]] when the mental model changes.

## Common Decisions

Use CSS when the DOM already has the behavior and you only need it to look different.

Use `.base` files when you need different columns, filters, formulas, visible fields, grouping, calendars, or boards.

Use TaskNotes frontmatter when the task data itself should change.

Use `kanban-bases-view` when the board is a generic Bases board and order can live inside the `.base` file.

Use `tasknotesKanban` when the board is made of TaskNotes tasks and order should live on task notes.

Use Context Nine when behavior needs JavaScript hooks into Obsidian, TaskNotes, DOM events, command registration, or file operations.

Use `vault bootstrap-export` when the change should apply to the public bootstrap vault. Use context folder scripts when the change should apply to local context folder creation.
